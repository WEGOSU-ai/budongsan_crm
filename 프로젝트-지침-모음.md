# 프로젝트 지침·설계 모음 (참고용)

> 부동산 매칭 CRM(매칭홈) 프로젝트에서 정립한 작업 지침·아키텍처·배포 절차를 한곳에 정리한 문서입니다. 인테리어 등 새 프로젝트의 출발 템플릿으로 재사용하세요. (작성 기준: 2026-05-30)

---

## 1. 제품 철학 / 작업 원칙

- **단일 HTML 파일**로 동작하는 무설치·무서버 앱을 기본형으로 한다. 더블클릭으로 열리고, 브라우저만 있으면 PC·모바일 어디서나 쓸 수 있어야 한다.
- **표면 기능보다 메커니즘**: "무엇을 보여줄까"보다 "데이터가 어디서 와서 어디에 저장되고 어떻게 동기화되는가"를 먼저 설계한다.
- **진단보다 실행**: 사용자가 다음 한 동작(번호만 고르면 되는 메시지, 한 번의 업로드, 한 번의 배포)으로 끝나게 만든다.
- **사실대로·과장 없이**: 법규(표시·광고법 등) 준수를 기능 안에 내장한다.

## 2. 기술 스택 / 아키텍처

- 프런트: 순수 HTML + Vanilla JS(빌드 도구 없음), 반응형 CSS(PC 상단탭 / 모바일 하단탭·+버튼).
- 외부 라이브러리는 **https CDN만** 사용: SheetJS(xlsx), html2canvas, jsPDF, (선택)Kakao SDK, @supabase/supabase-js.
- 데이터 저장은 **2계층**:
  1. 기본 = 브라우저 `localStorage`(기기 안, 오프라인 가능).
  2. 선택 = **Supabase(PostgreSQL)** 클라우드 동기화(다기기·다직원 공유).
- 설정 상수는 코드 상단에 모은다: `TODAY`, `KAKAO_JS_KEY`, `SUPABASE_URL`, `SUPABASE_KEY`.

## 3. 데이터 업로드 — "제목(머리글) 매칭" 원칙

엑셀/CSV 양식이 제각각이어도 **열 머리글(제목)만 맞으면 그 값을 인식**하게 만든다. 핵심 규칙:

- 열 순서가 바뀌어도, 일부 칸이 비어도, 한 시트에 여러 섹션이 쌓여 있어도 머리글을 자동 감지.
- 헤더 판정은 **정확한 제목 일치**(부분 문자열 매칭 금지 — 데이터 값이 헤더로 오인되는 버그 방지). 후보 제목 2개 이상 일치할 때만 헤더로 인정.
- 빈 잡행(3자리 이상 숫자가 없는 짧은 행)·섹션 제목은 자동 제외.
- 별칭(alias) 사전을 둬서 "단지명=건물명=매물명" 식으로 여러 표기를 한 필드로 흡수.
- 업로드 시 **추가/교체** 선택, 양식 다운로드(전체 컬럼 템플릿) 제공.

> 실전 교훈: 실제 거래처 양식(여러 벤더)으로 반드시 교차 검증할 것. 한 양식에서 196행→7행으로 누락된 버그는 "부분 문자열 헤더 매칭"이 원인이었다.

## 4. 매칭/스코어 엔진 (도메인 로직 예시)

- 점수 = 가격40% + 지역30% + 평형20% + 거래유형10%. 거래유형 다르면 제외.
- 금액(예산 ±10%) 안 맞으면 결과에서 **숨김**(전달 방해 방지).
- 도메인 보정: 월세는 환산보증금(보증금+월세×100)으로 비교, 아파트·토지는 평단가 계산.
- 중복(동일 연락처/매물) 1회로 정리, 1순위 강조 표시.

→ 인테리어 버전 응용: "고객 요구(평형·예산·스타일·일정) ↔ 시공 사례/견적 패키지" 매칭으로 치환 가능. 가중치·필수 제외 조건만 도메인에 맞게 재정의.

## 5. 보안 원칙 (점검 체크리스트)

- 시크릿/비밀번호 **하드코딩 금지**(공개키만 코드에, 비밀키는 서버/환경).
- 업로드 데이터의 화면 출력은 전부 **HTML escape**(XSS 차단). 마스킹 함수도 escape 처리.
- 개인정보 **마스킹 토글**(전화·이름·호수) 제공, 내보내기(이미지/PDF) 시 가린 채 저장.
- 데이터는 기본적으로 기기 내 저장(외부 전송 없음). 클라우드는 사용자가 로그인할 때만.
- `.gitignore`로 `.env`·키·백업파일 차단. 토큰은 파일에 저장하지 말고 실행 시 입력받기.
- 배포 시 `vercel.json`에 보안 헤더(X-Content-Type-Options, X-Frame-Options, Referrer-Policy) 적용.

## 6. 클라우드 동기화 설계 (Supabase)

- 테이블별 RLS 정책: `owner_id = auth.uid()` 로 본인 데이터만 접근.
- 인증: 이메일/비밀번호. 운영 편의를 위해 **이메일 자동 확인 트리거**(가입 즉시 사용 가능)로 설정 가능.
- **저장 시 자동 동기화** 패턴: 로컬 `save()` 호출 → 1.2초 디바운스 → 로그인 상태면 클라우드에 전체 교체 push. 헤더에 "☁ 저장됨 시각" 배지로 상태 표시.
- 다기기 흐름: 로그인 시 클라우드에 데이터 있으면 자동 pull(불러오기), 비어 있으면 현재 기기 데이터로 seed. 수동 올리기/내려받기 버튼도 병행.
- 계정 생성·비밀번호 입력은 **반드시 사용자 본인이** 한다(대행 금지).

## 7. 배포 — GitHub → Vercel 자동 배포 절차

1. GitHub에 빈 저장소 생성(예: `WEGOSU-ai/budongsan_crm`).
2. 파일 업로드: ① 로컬에서 `git push`(또는 github-push.bat), 또는 ② GitHub 웹 "Upload files"로 드래그 업로드.
3. Vercel 프로젝트 → Settings → Git → **Continue with GitHub** → (1회) Vercel GitHub 앱 권한 승인(조직·저장소 접근 허용) → 저장소 선택 → Connect.
4. 연결 후에는 저장소에 **push할 때마다 자동 배포**. 루트 `index.html` 자동 서비스, `vercel.json`이 헤더 적용.
5. 운영 도메인 확보 후 카카오/구글 OAuth 도메인 등록.

> 환경 제약 메모: 격리 샌드박스는 외부망이 막혀 `git push`·`vercel deploy`를 직접 못 한다. 실제 업로드/배포는 사용자 PC 또는 브라우저(웹 업로드)로 수행한다. Vercel/Supabase/브라우저 "커넥터"는 별도 망으로 동작하므로 조회·관리·DB 작업은 가능.

## 8. 공유 썸네일(og:image) / 메타태그

- `<head>`에 og:title/description/image, twitter:card=summary_large_image 추가.
- og:image는 1200×630 **PNG/JPG**(SVG는 대부분 SNS에서 미지원). 절대 URL로 지정.
- 한글 텍스트 이미지는 한글 폰트가 있는 환경(사용자 브라우저)에서 canvas로 렌더 → 다운로드해 저장소에 올린다.

## 9. 법규 준수 내장 (표시·광고법)

- 콘텐츠 생성 기능(블로그/SNS)은 법정 명시사항을 포함한 "최소 준수 틀(초안)"로 생성하고, 사실 확인·게시 책임을 사용자에게 명시.
- 사업자(중개사무소 등) 정보를 한 번 등록하면 모든 생성물 서명에 자동 삽입.

## 10. 인테리어 프로젝트 응용 포인트

- "매물↔고객" → "**시공 사례/견적 패키지 ↔ 상담 고객**" 으로 도메인 치환.
- 업로드 제목 매칭: 견적서·자재표·현장 체크리스트 등 제각각 양식을 머리글 기준으로 흡수.
- 필드 예시: 공간유형(아파트/상가/주택), 평형, 예산, 스타일(모던/내추럴…), 착공 희망일, 자재 등급, 현장 주소, 담당 시공팀, 진행단계(상담→실측→견적→계약→시공→AS).
- 파이프라인 단계와 자동 동기화·배포·보안·법규 내장 구조는 그대로 재사용.
- 차별 포인트: 사진(현장/포트폴리오) 다수 첨부·비교가 핵심이므로, 이미지 저장은 Supabase Storage 등 별도 설계 권장(localStorage 용량 한계).
